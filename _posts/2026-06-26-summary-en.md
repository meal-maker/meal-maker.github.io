## [Entire Herculaneum Scroll Deciphered Using AI for First Time](https://scrollprize.org/firstscroll) ⭐️ 10.0/10

For the first time, researchers have fully deciphered an entire Herculaneum scroll using AI-powered virtual unwrapping, recovering continuous ancient Greek text from the carbonized papyrus. The achievement was announced as part of the Vesuvius Challenge, with a preprint detailing the complete virtual unwrapping and reading of a rolled Herculaneum papyrus. This breakthrough demonstrates that AI can unlock texts from the only surviving library of the ancient world, buried and carbonized by the eruption of Mount Vesuvius in 79 CE. It opens the door to reading hundreds more scrolls without physically damaging them, potentially transforming our understanding of Greco-Roman philosophy and literature. The scroll was scanned using a particle accelerator (synchrotron) to generate high-resolution 3D CT scans, then AI models were trained to detect carbon-based ink on the carbonized papyrus substrate. The entire process, known as virtual unwrapping, involved segmenting the scroll layers and flattening them digitally—the complete text spans multiple columns.

hackernews · verditelabs · Jun 25, 15:48

**Background**: The Herculaneum scrolls are hundreds of papyrus rolls buried and carbonized by the eruption of Mount Vesuvius in 79 CE. They were excavated in the 18th century from a villa that likely belonged to the family of Julius Caesar's father-in-law. Because the carbon-based ink is indistinguishable from the carbonized papyrus using X-rays, traditional attempts to unroll them caused damage; AI and modern imaging now make it possible to read them non-invasively.

## [IBM debuts sub-1 nanometer chip technology](https://newsroom.ibm.com/2026-06-25-ibm-debuts-worlds-first-sub-1-nanometer-chip-technology) ⭐️ 8.0/10

IBM has announced the world's first sub-1 nanometer chip technology, a 0.7nm (7-angstrom) node called 'nanostack' that achieves higher transistor density through 3D stacking rather than traditional shrinking. This milestone demonstrates that transistor density can still be improved through innovative architectures like 3D stacking, even as traditional scaling faces physical limits. However, it also reinforces the disconnect between node naming and actual feature sizes, which affects how the industry communicates progress. The so-called 'sub-1nm' node name does not refer to any physical dimension on the chip; it is a marketing term that indicates approximately double the transistor density of the previous generation. IBM's technology uses a 3D stacked transistor architecture called 'nanostack' to achieve this density improvement.

hackernews · porridgeraisin · Jun 25, 15:33

**Background**: Historically, semiconductor process nodes were named after the gate length of transistors, but as scaling advanced, the numbers became decoupled from actual dimensions. By the late 2010s, node names like '7nm' no longer corresponded to any measurable feature and instead became shorthand for a generation of manufacturing technology. For example, a '5nm' node may have transistor features that are larger than 5nm. This background is necessary to understand that IBM's 'sub-1nm' claim is about density scaling, not literal atomic-scale transistors.

## [Zig introduces endian-agnostic bitCast and LLVM updates](https://ziglang.org/devlog/2026/#2026-06-25) ⭐️ 8.0/10

Zig has updated its @bitCast builtin to use endian-agnostic semantics, making operations like bitcasting [2]u8 to u16 behave identically on all targets, along with enhancements to the LLVM backend for better code generation. This change greatly simplifies writing portable code that manipulates binary data, as developers no longer need to handle endianness differences manually, and the LLVM enhancements improve runtime performance. Under the new semantics, bitCast only cares about logical bit representation, which is endian-agnostic, and arbitrary-width integer types like u4 and i13 are directly lowered to LLVM's bit-int types.

hackernews · kouosi · Jun 25, 14:19

**Background**: Endianness refers to the order of bytes in memory: little-endian stores the least significant byte first, big-endian stores the most significant byte first. Zig's @bitCast reinterprets the bits of a value as another type. Previously, bitcasting a small array to an integer would yield different results on big-endian vs. little-endian machines. The new semantics make this operation consistent across platforms, which is crucial for low-level code that reads binary protocols or file formats.

## [Un-0: Generating Images with Coupled Oscillators](https://unconv.ai/blog/introducing-un-0-generating-images-with-coupled-oscillators/) ⭐️ 7.0/10

The Un-0 project introduces a method to generate images by coupling oscillators, presenting it as an analog computing alternative to digital neural networks. The approach was demonstrated on a 64×64 pixel image, with the authors claiming potential energy efficiency gains. This work explores a fundamentally different computing paradigm for image generation, one that could drastically reduce power consumption if implemented in dedicated analog hardware. It also sparks important discussions about the scalability and practical viability of non-digital AI approaches. The method requires n² interconnections between oscillators, meaning a 4K image would need trillions of connections, posing a severe scaling challenge. Furthermore, the system was simulated on conventional digital hardware, so the claimed energy efficiency has not been experimentally validated in a physical implementation.

hackernews · babelfish · Jun 25, 20:50

**Background**: Analog computers use continuous physical quantities (e.g., voltage) to directly model problems, unlike digital computers that process discrete bits. Coupled oscillators are systems where multiple oscillating elements interact, often studied in physics and biology—for example, the Kuramoto model describes synchronization in such systems. The Un-0 work leverages synchronization patterns in oscillator arrays to encode image data, offering a potential path toward energy-efficient analog AI.

OS9Map aims to eliminate that need by implementing modern networking directly on the old operating system.

## [Go Project Applies Ousterhout's 'New Information' Decision Rule](https://colobu.com/2026/06/26/open-decision-making-from-go-issue/) ⭐️ 6.0/10

A blog post explains how the Go project uses John Ousterhout's 'What new information do you have?' principle to handle reconsideration of previously decided proposals. The principle, from Ousterhout's Open Decision-Making framework, requires a requester to present new information before re-opening a settled decision. This practice offers a clear, lightweight mechanism for managing recurring debates in large open-source projects, helping maintainers avoid endless re-discussions while remaining open to legitimate updates. It serves as a practical model for other communities seeking to balance openness with decision finality. The Go proposal process explicitly cites the Reconsideration section of Ousterhout's Open Decision-Making article as its guideline. Ousterhout's framework includes four steps: collect input widely, facilitate consensus, make the decision, and document the reasoning.

rss · 鸟窝 · Jun 26, 00:00

**Background**: The Go project receives numerous proposals daily, many revisiting previously discussed and decided topics, which consumes maintainer time. John Ousterhout is a Stanford professor known for his book 'A Philosophy of Software Design' and for developing the Tcl scripting language. His Open Decision-Making framework advocates for collective input while maintaining decision efficiency, challenging the view that open processes are necessarily slow or chaotic.

## [Go's 'seq' Library Brings Scala-Like Chained Pipelines](https://colobu.com/2026/06/25/go-seq-chained-pipeline-like-scala/) ⭐️ 6.0/10

A new Go library called 'seq' by the user smallnest enables chained pipeline operations that read left-to-right, similar to Scala's collection pipelines; the library was built in 2 hours with AI assistance using the GLM-5.2 model and demonstrated in a live stream. This library offers a more natural and readable way to compose map, filter, and other functional operations in Go, which could improve developer productivity and appeal to programmers from functional programming backgrounds like Scala or Clojure. The development used Loop Engineering with Volcengine's coding plan, consuming 7.23M tokens over 2 hours; the library's source code and design documents are available on GitHub at github.com/smallnest/seq under the 'tasks' directory.

rss · 鸟窝 · Jun 25, 10:38

**Background**: In Go, functional operations often require nested calls like lo.Map(lo.Filter(...)), which are less readable. Libraries like samber/lo provide type-safe generics-based helpers for map, filter, and reduce. A pipeline pattern chains functions sequentially, and the new 'seq' library aims to improve that experience by offering a left-to-right chain syntax similar to Scala's collection pipelines.
